## End-to-end runtime path (real code, today)

### Path-1 (user has a Sui wallet — PTB-direct)

```
Integrator → POST /v1/sender/orders
  → SenderController persists PaymentOrder + Recipient
  → Returns pay_to.options.ptb (built client-side or by integrator UI)
User signs PTB in their Sui Wallet / Suiet / zkLogin
  → tx executes on-chain: create_order moves coin into Gateway escrow
SuiEventIndexer (WebSocket)
  → OrderCreated event arrives sub-second
  → handleOrderCreated → priorityQueue.AssignLockPaymentOrder
LP picks up order via webhook (matching engine round-robin)
LP fulfills fiat via BaaS (out of scope for our backend; BaaS calls Rails on success)
Aggregator submits settle_order PTB (OrderSui.SettleOrder)
  → On-chain: Order escrow drained to LP wallet
SuiEventIndexer
  → OrderSettled event
  → handleOrderSettled: DB transaction (TransactionLog + LockPaymentOrder + PaymentOrder) + integrator webhook
```

### Path-2 (user pays from Bybit / Binance / OKX — exchange withdrawal)

Same as Path-1, but two extra steps after order creation, before `OrderCreated`:

```
Integrator → POST /v1/sender/orders
  → SenderController generates Sui receive address (Ed25519 keypair)
  → Persists SuiReceiveAddress entity (status=unused)
  → Returns address + QR to integrator → user
User pastes address into exchange withdraw form, sends USDC
  → Coin lands at receive address on Sui
SuiDepositWatcher (every 1 min cron)
  → sui_getCoins detects deposit
  → SuiReceiveAddress.status = deposited
  → OrderSui.CreateOrder: sponsored PTB
     → Receive wallet signs user payload
     → Aggregator pays gas
     → MoveCall: rails::order::create_order(...) forwards coin into Gateway escrow
  → SuiReceiveAddress.status = forwarded
[from here, same as Path-1: SuiEventIndexer sees OrderCreated, matching, etc.]
```

## Architectural rules (don't break)

1. **No Rails-controlled wallet holds user funds except for in-flight bridging (Route A) and Path-2 receive-address custody (seconds).** Any design proposing a "Rails treasury account for user funds" is a custody slip — stop.
2. **No per-payout-address proof of ownership.** Matches the established pattern: KYC-time wallet signature proves ONE wallet, subsequent payout addresses are declared freely (controllers/accounts/profile.go pattern). Don't tighten this without explicit reason.
3. **LPs are passive.** No webhook server, no node. They deposit fiat into a BaaS virtual account, configure rate band in dashboard, done. The `host_identifier` field on `ProviderProfile` is a leftover from the legacy polling model — leave it nil for Sui LPs.
4. **The Move Gateway is the source of truth for on-chain state.** DB mirrors via the indexer. Never write status transitions to `LockPaymentOrder` from anywhere except the `SuiEventIndexer` handlers (or `OrderSui.SettleOrder` /  `RefundOrder` which trigger those events).
